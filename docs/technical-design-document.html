2. Data Flow

Each pipeline execution follows a strict 12-step sequence. Steps 1–6 run for each of the four source workers, and steps 7–12 run once across all sources.

PHASE 1: Per-Worker Ingestion (×4 sources) ═══════════════════════════════════════════ ① ArcGIS Query → Paginated fetch (1000 records/page, 1.5s delay) ② Keyword Filter → Permit type + description matching ③ Deduplication → By permit_number, then by address+date hash ④ Score Compute → Apply scoring rules (DB-backed, 5-min cache) ⑤ Geocode → US Census → Nominatim/OSM fallback ⑥ DB Upsert → Batch insert/update to Supabase + mirror PHASE 2: Post-Ingestion Enrichment (×1, all sources) ═══════════════════════════════════════════════════════ ⑦ Geocode Maint → Fill remaining NULL lat/lon (cap: 200/run) ⑧ DERM Addr Fix → Re-query ArcGIS for DERM leads missing address ⑨ PA Lookup → Miami-Dade Property Appraiser: owner, mailing addr ⑩ Enformion → Paid API: address/contact for DERM leads (cache-first) ⑪ Contact Enrich → Multi-source cascade: person → business → Sunbiz → trust → property ⑫ Re-Score → Recompute all leads with current scoring rules PHASE 3: Reporting (conditional) ═════════════════════════════════ ⑬ Email Digest → If SEND_EMAIL=true AND total_inserted > 0

Incremental Query Strategy

On each run, workers query only records newer than the last successful run's finished_at timestamp (stored in job_runs table). This prevents re-processing the entire dataset.

4. Orchestrator (pipeline/main.py)

The main module coordinates the entire pipeline execution. When invoked with --once (standard for GitHub Actions), it:

  1. Creates an overarching job_run record (pipeline_sync)
  2. Runs all 4 source workers sequentially (DERM → Fort Lauderdale → Miami Tree → Miami Building)
  3. Catches and logs exceptions per worker (one failure doesn't stop the pipeline)
  4. Runs geocode maintenance for all sources
  5. Runs DERM address re-check against ArcGIS
  6. Runs property enrichment (PA lookup)
  7. Runs Enformion enrichment (cache-first, then API for new leads)
  8. Runs contact enrichment (multi-source cascade)
  9. Re-scores all leads with current DB scoring rules
  10. Completes the overarching job_run with summary stats
  11. Sends email digest if SEND_EMAIL=true AND total_inserted > 0

Error Isolation

Each worker and enrichment step is wrapped in a try/except block. A failure in one step logs the error and continues to the next. The overarching job_run status is set to partial if any source failed, success otherwise.

Email Guard

Email digests are never sent when total_inserted == 0. This is a hard rule that prevents empty notification spam. The environment flag SEND_EMAIL must also be explicitly set to true.

5. Source Workers

All four workers share the same abstract contract:

def run_worker() -> Dict[str, Any]:
    """Returns: {status, records_found, records_inserted, records_updated, errors}"""

Each worker:

  1. Creates its own job_run record
  2. Queries its ArcGIS endpoint with incremental filters
  3. Applies keyword filtering (if general permit dataset)
  4. Normalizes records to a common lead schema
  5. Deduplicates against existing DB records
  6. Computes lead scores
  7. Batch upserts into Supabase
  8. Completes its job_run with results

5.1 DERM Worker (pipeline/workers/derm.py)

Date Interpolation Algorithm

DERM's ArcGIS dataset has no date field. The pipeline estimates permit dates using sequence number interpolation:

Anchor point: TREE-02600309 → 2026-04-04 (Friday) Rate: ~4.61 tree permits per business day Method: Walk calendar forward/backward from anchor by (target_seq - anchor_seq) / 4.61 business days Sequence structure: WORK_GROUP_NUMBER = YYNNNNN (7 digits) YY = 2-digit year prefix (26 → 2026) NNNNN = sequence within year (resets each January) Annual volumes: 2024: 1,105 permits → 4.24/day 2025: 1,384 permits → 5.32/day 2026: ~1,187 permits → 4.61/day (current estimate) Special cases: - Pre-2024 records → YYYY-01-01 fallback - NEW real-time permits → actual discovery date (no interpolation) - Business days exclude weekends + 11 federal holidays

Backfill Floor

DERM_BACKFILL_MIN_WGN = 2500001  # First 2025 permit
# Records before this are NEVER fetched or inserted
# Update to 2600001 in January 2027

Incremental Strategy

Uses WORK_GROUP_NUMBER > highest_wgn_in_db as the WHERE clause. No date-based filtering.

6. Filtering & Classification (pipeline/filters.py)

A 4-step filtering pipeline determines whether a permit record is tree/vegetation-related:

Step 1: Exact Permit Category Match

EXACT_PERMIT_TYPES = {
    "TREE REMOVAL", "TREE REMOVAL PERMIT", "ARBOR PERMIT",
    "VEGETATION REMOVAL", "TREE ALTERATION",
    "LANDSCAPE INSTALLATION", "ROW LANDSCAPING PERMIT",
    "LANDSCAPE PERMIT"
}

Step 2: Description Keyword Match

Positive keywords: tree, arbor, vegetation, landscape, canopy, removal, trimming, cutting, etc.

Step 3: Negative Keyword Exclusion

Reject records matching: fence, sign, shed, pool, solar, hvac, roof, etc. These are common false positives in general building permit datasets.

Step 4: Data Quality Validation

  • Coordinates within South Florida bounding box (lat 25.1–26.4, lon −80.9 to −80.0)
  • Non-null address or non-null permit date
  • Permit age within MAX_PERMIT_AGE_DAYS

7. Deduplication (pipeline/dedup.py)

Two-layer deduplication prevents duplicate leads:

Layer 1: Permit Number Dedup

Check permit_number against existing records. For City of Miami permits, suffixes like L001 are stripped before comparison (e.g., BD26003591001L001 → BD26003591001).

Layer 2: Address + Date Hash Dedup

If a permit number appears new, a secondary check compares normalized_address + permit_date. This catches re-issued permits where the same property gets a new permit number for the same work.

8. Scoring Engine

Architecture (pipeline/scoring.py + pipeline/scoring_config.py)

┌─────────────────────┐ ┌──────────────────────┐ │ scoring_config.py │ │ scoring_rules table │ │ ┌───────────────┐ │ │ (Supabase) │ │ │ DEFAULTS dict │ │◄────│ Singleton row │ │ │ (hardcoded) │ │ │ CHECK (id = 1) │ │ └───────┬───────┘ │ └──────────┬───────────┘ │ │ fallback │ │ primary │ ▼ │ ▼ │ get_scoring_rules() │───► Cached rules (5-min TTL) └──────────┬──────────┘ ▼ compute_lead_score(lead, rules) → int (0–10)

Score Computation Flow

  1. Start with base score = 0
  2. Apply permit type bonuses (+1 to +5)
  3. Apply recency bonuses (source-specific tiers)
  4. Apply property characteristic bonuses (large parcel, contact info)
  5. Apply penalties (no address, contractor assigned, bad status, stale)
  6. Clamp result to 0–10 range

Re-Scoring (pipeline/rescore.py)

At the end of every pipeline run, all leads are re-scored in batches of 500. This ensures:

  • Recency bonuses decay correctly as leads age
  • Admin scoring rule changes in the dashboard take effect
  • Only leads whose score actually changed are updated (avoids unnecessary DB writes)

Results outside this box are rejected (indicates bad geocoding — e.g., a "Miami" match in Ohio).

Maintenance (pipeline/geocode_maintenance.py)

A source-agnostic batch job fills NULL lat/lon for all sources:

  • Default cap: 200 addresses per run
  • Non-blocking: errors logged and skipped
  • Idempotent: only touches rows with latitude IS NULL and non-empty address

Search Types

search_person()

Person search by name + state + optional address. Returns phone, email, address matches. Used for individual property owners.

search_business()

Business search by company name + state. Returns business phone, registered agent, officers. Used for LLC/Corp property owners.

search_property()

Property/address reverse lookup using only AddressLine1, City, State. Returns current/previous occupant contact details. Used as last-resort fallback when name-based searches fail.

Git-Backed JSON Cache

address-contact-data/ ├── TREE-02600001.json ← Full Enformion API response ├── TREE-02600002.json ├── TREE-02600003.json └── ... Strategy: 1. Check if address-contact-data/{permit_number}.json exists 2. If YES → read from disk (FREE, no API call) 3. If NO → call Enformion API ($0.25) → save JSON → commit to git 4. On next run: step 1 finds the cached file → no cost The GitHub Actions workflow commits new JSON files back to the repo, so the cache grows over time and is shared across runs.

12. Contact Enrichment Cascade (pipeline/contact_enrichment.py)

This module extends Enformion enrichment to cover all 4 lead sources with a multi-step fallback chain:

FOR EACH LEAD (last 30 days, missing owner_phone): ═══════════════════════════════════════════════════ Step 1: Cache Check └─ Check address-contact-data/{permit}.json └─ Found phone? → DONE (free) Step 2: Enformion Person Search └─ Query: FirstName + LastName + State + Address └─ Found phone? → DONE ($0.25) Step 3: Enformion Business Search └─ If owner_name looks like LLC/Corp/Inc └─ Query: CompanyName + State └─ Found phone? → DONE ($0.25) Step 4: Sunbiz Officer Lookup └─ If FL business, scrape registered officers from sunbiz.org └─ Re-run person search with officer name └─ Found phone? → DONE ($0.25) Step 5: Trust Name Parsing └─ If owner_name contains "TRUST", "TRUSTEE", "REVOCABLE" └─ Extract person name from trust entity name └─ Re-run person search with extracted name └─ Found phone? → DONE ($0.25) Step 6: Property Reverse Search └─ LAST RESORT — search by address only (no name) └─ Returns current/previous occupant contacts └─ Found phone? → DONE ($0.25) NOT FOUND → Mark enformion_checked_at to prevent re-query

Contractor Enrichment

Separately, for leads with contractor_name but no contractor_phone:

  1. Check cache for contractor name match
  2. Enformion business search by contractor company name
  3. Save result to cache and update DB

14. Dual-Write Architecture

When both SECONDARY_SUPABASE_URL and SECONDARY_SUPABASE_KEY are configured, all enrichment writes are automatically mirrored to a secondary Supabase instance.

Pipeline Write Flow: ┌──────────┐ │ Pipeline │──► db.update_lead(data) ──┬──► Primary Supabase ✅ │ Module │ │ └──────────┘ └──► Secondary Supabase (best-effort) └─ Failure? → Log warning, continue └─ NEVER blocks primary pipeline

Design principle: The secondary DB is a convenience for keeping dev and prod in sync. A failure writing to secondary is logged but never blocks the primary pipeline or raises an exception.

18. Dashboard Frontend

The dashboard is a single-page application (SPA) built with vanilla HTML, CSS, and JavaScript — no framework dependencies. It's hosted as a static site on GitHub Pages.

  • Actions: Lead review, email subscription, manual dispatch all route through Cloudflare Worker

Distributed advisory locks via the pipeline_locks Supabase table prevent dangerous concurrent operations:

from pipeline.pipeline_lock import PipelineLock

with PipelineLock("backfill", ttl_seconds=7200) as lock:
    if lock.acquired:
        do_backfill_work()
    else:
        print(f"Blocked by: {lock.blocker}")
        # Lock auto-expires after TTL if holder crashes

24. Security Model

Secret Management

LayerStorageAccess
Pipeline (Python)GitHub Actions SecretsInjected as env vars at runtime
Dashboard (JS)Supabase anon key (public, RLS-protected)Visible in source — read-only by design
API MiddlewareCloudflare Worker SecretsEncrypted at rest, available only in Worker runtime
Database AdminSupabase service_role keyPipeline and Worker only — never exposed to frontend

Authentication Flow

Browser → POST /api/auth (username, password) → Cloudflare Worker validates against DASHBOARD_USERNAME/PASSWORD secrets → Returns session token → Browser stores in sessionStorage (expires on tab close) → Subsequent API calls include token in Authorization header

Database Security

  • Row Level Security (RLS): Supabase RLS policies control read/write access
  • Anon key: Dashboard uses anon key for read-only PostgREST queries
  • Service role key: Used by pipeline and Worker for write operations — never exposed to frontend

Source Code Security

  • Zero hardcoded credentials in production code
  • All secrets stored in GitHub Secrets / Cloudflare Worker secrets
  • .env files excluded via .gitignore
  • Production repo guard prevents accidental paid API usage from dev environments
